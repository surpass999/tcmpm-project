# 审批流程节点、参与角色与流转条件分析

## 一、需求概述

根据需求文档3.11节，业务流程包括5个核心流程：
1. 备案流程
2. 中期评估流程
3. 交流流程
4. 验收流程
5. 转化流程
6. 整改流程

---

## 二、各审批流程详细分析

### 2.1 备案流程

**流程定义**："项目单位申报 → 省级审核 → 专家论证 → 国家备案"

#### 节点设计

| 序号 | 节点名称 | 节点类型 | 说明 |
|------|----------|----------|------|
| 1 | 项目单位申报 | 发起人节点 | 项目单位编制并提交建设方案 |
| 2 | 省级审核 | 审批人节点 | 省级中医药主管部门受理审核 |
| 3 | 专家论证 | 审批人节点 | 国家层面组织专家开展论证评审 |
| 4 | 国家备案 | 结束节点 | 备案确认，形成备案结论 |

#### 参与角色

| 角色 | 说明 | 对应系统配置 |
|------|------|--------------|
| 项目单位 | 申报发起人 | 发起人节点（START_USER_NODE） |
| 省级中医药主管部门 | 省级审核人 | 审批人节点（角色：省级审核角色） |
| 专家组 | 论证评审人 | 审批人节点（用户组：专家库） |
| 国家管理端 | 备案确认 | 结束节点 |

#### 流转条件

```
项目单位申报 
    ↓ (提交)
省级审核
    ↓ (审核通过)
    ↓ (审核不通过，退回完善)
专家论证
    ↓ (论证通过)
    ↓ (需整改，反馈项目单位完善)
国家备案 ← 备案确认，流程结束
```

**条件分支**：
- 省级审核：条件分支（通过/退回）
- 专家论证：条件分支（通过/需整改）

#### BPM系统支持度评估

| 能力 | 支持程度 | 说明 |
|------|----------|------|
| 审批节点 | ✅ 完全支持 | APPROVE_NODE 支持多人员审批方式 |
| 审批人分配 | ✅ 完全支持 | 支持角色、用户组、用户等多种策略 |
| 条件分支 | ✅ 完全支持 | CONDITION_BRANCH_NODE 支持 |
| 驳回处理 | ✅ 完全支持 | 支持RejectHandler配置驳回到指定节点 |
| 多人审批方式 | ✅ 完全支持 | 支持会签、按比例通过等 |

### 2.2 中期评估流程

**流程定义**："项目单位自评 → 省级审核 → 专家评估 → 整改落实"

#### 节点设计

| 序号 | 节点名称 | 节点类型 | 说明 |
|------|----------|----------|------|
| 1 | 项目单位自评 | 发起人节点 | 提交中期自评报告及支撑材料 |
| 2 | 省级审核 | 审批人节点 | 省级主管部门审核材料 |
| 3 | 专家评估 | 审批人节点 | 国家层面组织专家开展评估 |
| 4 | 整改落实 | 办理人节点 | 项目单位按要求整改 |
| 5 | 整改复核 | 审批人节点 | 专家对整改结果复核确认 |
| 6 | 评估结论 | 结束节点 | 形成最终中期评估结论 |

#### 参与角色

| 角色 | 说明 | 对应系统配置 |
|------|------|--------------|
| 项目单位 | 自评发起人/整改人 | 发起人节点 + 办理人节点 |
| 省级中医药主管部门 | 审核人 | 审批人节点（角色：省级审核角色） |
| 专家组 | 评估人/复核人 | 审批人节点（用户组：专家库） |

#### 流转条件

```
项目单位自评
    ↓ (提交)
省级审核
    ↓ (审核通过)
    ↓ (审核不通过，退回修改)
专家评估
    ↓ (评审通过)
    ↓ (需整改)
整改落实
    ↓ (提交整改说明与证据)
整改复核
    ↓ (复核通过)
评估结论 ← 流程结束
```

**关键特性**：
- 存在"整改落实"循环：评估不通过 → 整改 → 复核 → 可能再次整改
- 需要支持多次驳回和回归

#### BPM系统支持度评估

| 能力 | 支持程度 | 说明 |
|------|----------|------|
| 审批节点 | ✅ 完全支持 | APPROVE_NODE |
| 办理人节点 | ✅ 完全支持 | TRANSACTOR_NODE 支持处理人配置 |
| 循环审批 | ✅ 支持 | 通过驳回（RejectHandler）实现 |
| 表单关联 | ✅ 支持 | 可配置表单与流程关联 |

### 2.3 整改机制（贯穿性机制）

**重要说明**：整改不是独立的流程，而是一个**贯穿多个审批流程的闭环管理机制**。它存在于备案流程、中期评估流程、验收流程等多个环节。

#### 整改机制涉及的业务场景

| 序号 | 业务场景 | 整改发起方 | 整改执行方 | 整改确认方 |
|------|----------|------------|------------|------------|
| 1 | 备案流程-专家论证 | 专家组 | 项目单位 | 专家组 |
| 2 | 中期评估流程-评估反馈 | 专家组 | 项目单位 | 专家组 |
| 3 | 验收流程-省级审核 | 省级主管部门 | 项目单位 | 省级主管部门 |
| 4 | 验收流程-省级验收 | 专家组 | 项目单位 | 专家组 |
| 5 | 验收流程-国家复核 | 国家管理端 | 项目单位 | 国家管理端 |
| 6 | 风险预警整改 | 省级/国家管理端 | 项目单位 | 省级/国家管理端 |

#### 整改流程节点设计

| 序号 | 节点名称 | 节点类型 | 说明 |
|------|----------|----------|------|
| 1 | 整改下发 | 审批人节点 | 审批人填写整改要求，驳回到项目单位 |
| 2 | 整改填写 | 办理人节点 | 项目单位提交整改说明和支撑材料 |
| 3 | 整改复核 | 审批人节点 | 审批人复核整改结果 |

#### 参与角色

| 角色 | 说明 | 对应系统配置 |
|------|------|--------------|
| 审批人 | 下发整改要求，填写整改清单 | 审批人节点（ APPROVE_NODE） |
| 项目单位 | 填写整改说明，提交整改材料 | 办理人节点（TRANSACTOR_NODE） |
| 专家/主管部门 | 复核整改结果，确认整改完成 | 审批人节点（ APPROVE_NODE） |

#### 整改流转条件

```
审批环节
    ↓ (评估/审核不通过，需整改)
整改下发 ← 审批人填写整改清单，明确整改要求和时间
    ↓ (驳回到项目单位)
整改填写 ← 项目单位提交整改说明与支撑材料
    ↓ (提交整改结果)
整改复核 ← 审批人复核整改结果
    ↓ (复核通过)
后续环节继续
    ↓ (复核不通过，需再次整改)
整改填写 ← 循环回到项目单位继续整改
```

#### 整改闭环的关键特性

1. **多层级整改**：可存在于审批流程的任意审批节点之后
2. **循环机制**：支持"整改→复核→再次整改"的多次循环
3. **证据链管理**：整改说明与原始问题对应，支撑材料与整改要求绑定
4. **进度跟踪**：整改过程中的进展情况持续更新
5. **档案归集**：整改结果纳入项目过程管理档案

#### 整改任务分解与跟踪

根据需求文档3.4.3节，整改机制还包括：
- **任务分解**：将整改事项分解为具体任务
- **责任落实**：明确整改责任人和完成时限
- **进度跟踪**：实时更新整改进展状态
- **复核确认**：整改完成后由审批人确认

#### BPM系统支持度评估

| 能力 | 支持程度 | 说明 |
|------|----------|------|
| 驳回处理 | ✅ 完全支持 | RejectHandler 支持驳回到指定节点 |
| 循环审批 | ✅ 完全支持 | 支持多次整改循环 |
| 表单关联 | ✅ 完全支持 | 整改表单与原始问题关联 |
| 任务跟踪 | ⚠️ 需扩展 | 需配合业务模块实现任务分解和进度跟踪 |
| 超时处理 | ✅ 支持 | TimeoutHandler 支持整改时限提醒 |
| 消息通知 | ✅ 支持 | 触发器节点支持整改通知 |

#### 整改机制实现建议

1. **通用整改子流程**：设计可复用的整改子流程（CHILD_PROCESS），嵌入到各审批流程中
2. **整改表单模板**：预定义整改申请表单，包含整改说明、支撑材料上传、整改计划等字段
3. **整改任务模块**：独立的任务管理模块，支持任务分解、责任落实、进度跟踪
4. **超时提醒**：配置TimeoutHandler，对超时未完成整改进行提醒
5. **档案关联**：整改记录与原始审批流程归档关联

### 2.4 交流流程

**流程定义**："活动组织 → 材料沉淀 → 经验复用"

#### 节点设计

| 序号 | 节点名称 | 节点类型 | 说明 |
|------|----------|----------|------|
| 1 | 活动组织 | 发起人节点 | 国家/省级组织交流活动 |
| 2 | 材料归集 | 办理人节点 | 活动材料与成果集中归集 |
| 3 | 整理归档 | 办理人节点 | 整理形成标准化经验材料 |
| 4 | 经验入库 | 结束节点 | 纳入经验库与工具包 |

#### 参与角色

| 角色 | 说明 | 对应系统配置 |
|------|------|--------------|
| 国家/省级管理员 | 活动组织发起人 | 发起人节点 |
| 工作人员 | 材料归集/整理 | 办理人节点（TRANSACTOR_NODE） |

#### 流转条件

```
活动组织
    ↓ (发布活动信息、组织报名)
材料归集
    ↓ (归集会议资料、演示成果)
整理归档
    ↓ (整理形成标准化材料)
经验入库 ← 流程结束
```

**特点说明**：
- 此流程为信息归集类，非严格审批流
- 节点之间为串行处理
- 可简化为普通流程

#### BPM系统支持度评估

| 能力 | 支持程度 | 说明 |
|------|----------|------|
| 发起人节点 | ✅ 完全支持 | START_USER_NODE |
| 办理人节点 | ✅ 完全支持 | TRANSACTOR_NODE |
| 串行流程 | ✅ 完全支持 | 简单顺序结构 |

### 2.5 验收流程

**流程定义**："项目单位申请 → 省级审核/验收 → 国家复核确认"

**特点**：支持灵活配置，可选择国家或省级组织验收

#### 节点设计（省级验收模式）

| 序号 | 节点名称 | 节点类型 | 说明 |
|------|----------|----------|------|
| 1 | 验收申请 | 发起人节点 | 项目单位提交验收申请与材料 |
| 2 | 省级审核 | 审批人节点 | 省级审核验收申请和材料 |
| 3 | 省级验收 | 审批人节点 | 省级组织验收 |
| 4 | 国家复核 | 审批人节点 | 国家层面复核确认 |
| 5 | 结果发布 | 结束节点 | 发布验收结论，完成成果认定 |

#### 参与角色

| 角色 | 说明 | 对应系统配置 |
|------|------|--------------|
| 项目单位 | 申请发起人 | 发起人节点 |
| 省级中医药主管部门 | 审核人/验收组织者 | 审批人节点 |
| 专家组 | 验收评审人 | 审批人节点（用户组：专家库） |
| 国家管理端 | 复核确认 | 审批人节点 + 结束节点 |

#### 流转条件

```
验收申请
    ↓ (提交申请与材料)
省级审核
    ↓ (审核通过)
    ↓ (需补充整改，退回完善)
省级验收
    ↓ (验收通过)
    ↓ (验收不通过，下发整改要求)
国家复核 ← 报送国家管理端
    ↓ (复核确认)
结果发布 ← 验收结论发布，流程结束
```

**条件分支**：
- 省级审核：条件分支（通过/需补充整改）
- 省级验收：条件分支（通过/不通过，需整改）
- 验收模式：路由分支（省级验收/国家直接验收）

#### BPM系统支持度评估

| 能力 | 支持程度 | 说明 |
|------|----------|------|
| 审批节点 | ✅ 完全支持 | APPROVE_NODE |
| 路由分支 | ✅ 完全支持 | ROUTER_BRANCH_NODE 支持多模式配置 |
| 条件分支 | ✅ 完全支持 | CONDITION_BRANCH_NODE |
| 评分规则 | ⚠️ 需扩展 | 需配合表单和业务逻辑实现 |
| 结论归档 | ✅ 支持 | 通过触发器节点实现 |

### 2.6 转化流程

**流程定义**："成果遴选 → 转化形成 → 推广应用"

#### 节点设计

| 序号 | 节点名称 | 节点类型 | 说明 |
|------|----------|----------|------|
| 1 | 成果汇集 | 发起人节点 | 汇集验收通过项目的建设成果 |
| 2 | 成果遴选 | 审批人节点 | 按标准筛选示范成果 |
| 3 | 成果转化 | 办理人节点 | 转化为规范/模式/典型案例 |
| 4 | 发布推广 | 结束节点 | 平台统一发布，行业复制与示范推广 |

#### 参与角色

| 角色 | 说明 | 对应系统配置 |
|------|------|--------------|
| 系统管理员 | 成果汇集 | 自动触发或发起人节点 |
| 主管部门 | 遴选审批 | 审批人节点 |
| 工作人员 | 成果转化 | 办理人节点 |

#### 流转条件

```
成果汇集 ← 来自验收通过项目
    ↓ (自动/手动触发)
成果遴选
    ↓ (筛选通过)
成果转化
    ↓ (转化完成)
发布推广 ← 流程结束
```

**特点说明**：
- "成果汇集"可能为自动触发（集成验收流程）
- 此流程偏重信息处理，审批环节较少

#### BPM系统支持度评估

| 能力 | 支持程度 | 说明 |
|------|----------|------|
| 自动触发 | ⚠️ 需扩展 | 需通过TRIGGER_NODE或流程事件实现 |
| 审批节点 | ✅ 完全支持 | APPROVE_NODE |
| 办理人节点 | ✅ 完全支持 | TRANSACTOR_NODE |
| 子流程 | ✅ 完全支持 | CHILD_PROCESS 可嵌入其他流程 |

## 四、BPM系统能力评估总结

### 4.1 完全支持的流程能力

| 能力 | 说明 |
|------|------|
| 多种节点类型 | 开始/结束、发起人、审批人、抄送人、办理人、延迟器、触发器、子流程、条件/并行/包容/路由分支 |
| 审批人分配策略 | 角色、部门、岗位、用户、用户组、表达式、表单字段等12种策略 |
| 多人审批方式 | 顺序审批、会签、按比例通过、并行审批 |
| 条件流转 | 支持表达式配置条件分支 |
| 驳回处理 | 支持驳回到指定节点 |
| 超时处理 | 支持超时自动通过/转交/提醒 |
| 任务监听器 | 支持创建/指派/完成等事件触发 |

### 4.2 需要扩展的能力

| 能力 | 当前限制 | 建议方案 |
|------|----------|----------|
| 评分规则 | 无内置评分规则 | 通过表单+业务逻辑实现，配合Trigger节点 |
| 评分汇总 | 无内置汇总 | 通过流程变量+业务服务实现 |
| 自动触发 | 需配置Trigger节点 | 预定义触发器模板，供流程调用 |
| 连续多级审批 | 需手动配置多个节点 | 封装审批人选择策略模板 |

### 4.3 流程配置建议

1. **备案流程**：标准审批流，使用条件分支处理审核不通过情况
2. **中期评估流程**：带循环的审批流，使用驳回机制实现整改循环
3. **交流流程**：简化为信息处理流，使用办理人节点串行处理
4. **验收流程**：支持路由分支切换省级/国家验收模式
5. **转化流程**：信息处理流，可集成触发器实现自动汇集

## 五、结论

根据对需求文档和现有BPM模块能力的分析：

1. **所有5个核心审批流程均可在现有BPM模块中实现**
2. **整改机制为贯穿性机制**：整改不是独立流程，而是嵌入在备案、中期评估、验收等多个流程中的闭环管理机制
3. **主要审批节点类型**：发起人节点、审批人节点、办理人节点、条件分支节点、路由分支节点
4. **主要流转条件**：通过/不通过分支、驳回重审、整改循环、多模式路由
5. **角色配置**：通过角色、用户组、用户、表达式等多种策略实现
