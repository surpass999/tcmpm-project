# SIMPLE 设计器 审批流程类型清单与缺口分析

## 一、审批流程类型清单

### 1.1 核心审批流程（5个）

| 序号 | 流程名称 | 流程定义 | 涉及角色 |
|------|----------|----------|----------|
| 1 | 备案流程 | 项目单位申报 → 省级审核 → 专家论证 → 国家备案 | 项目单位 → 省级 → 专家 → 国家 |
| 2 | 中期评估流程 | 项目单位自评 → 省级审核 → 专家评估 → 整改落实 | 项目单位 → 省级 → 专家 |
| 3 | 交流流程 | 活动组织 → 材料沉淀 → 经验复用 | 国家/省级管理员 → 工作人员 |
| 4 | 验收流程 | 项目单位申请 → 省级审核/验收 → 国家复核确认 | 项目单位 → 省级 → 国家 |
| 5 | 转化流程 | 成果遴选 → 转化形成 → 推广应用 | 主管部门 → 工作人员 |
| 6 | 整改流程 | 业务异常、指标异常、评审不通过 → 业务部门整改 → 发起人评审 |


### 1.2 审批业务类型明细

| 序号 | 业务模块 | 审批类型 | 对应流程 | 涉及角色 |
|------|----------|----------|----------|----------|
| 1 | 建设方案备案管理 | 备案申报审批 | 备案流程 | 项目单位 → 省级 → 专家 → 国家 |
| 2 | 省级审核与汇交 | 省级审核 | 备案流程 | 省级中医药主管部门 |
| 3 | 专家论证与备案归档 | 专家评审 | 备案流程 | 专家组 |
| 4 | 中期评估管理 | 自评+专家评估 | 中期评估流程 | 项目单位 → 省级 → 专家 |
| 5 | 中期评估管理 | 评估反馈与整改跟踪 | 整改机制（贯穿） | 专家组 → 项目单位 → 专家组 |
| 6 | 验收与结果管理 | 验收审批 | 验收流程 | 项目单位 → 省级 → 国家 |
| 7 | 验收与结果管理 | 验收整改跟踪 | 整改机制（贯穿） | 验收方 → 项目单位 → 验收方 |
| 8 | 推广应用与成果转化 | 成果遴选 | 转化流程 | 主管部门 |
| 9 | 专家库与评审支撑 | 评审任务派发 | 各流程通用 | 项目管理员 |
| 10 | 风险预警管理 | 风险整改管控 | 整改机制（贯穿） | 省级/国家 → 项目单位 → 省级/国家 |

## 六、流程启动触发机制

### 6.1 触发类型概述

根据业务需求，审批流程的启动主要有以下两种触发方式：

| 触发类型 | 触发条件 | 典型场景 | 支持状态 |
|----------|----------|----------|----------|
| 用户提交触发 | 用户在业务模块提交数据 | 备案申报、验收申请 | ✅ 完全支持 |
| 定时器触发 | 达到预设时间条件 | 半年报、年报自动触发 | ✅ 完全支持 |

### 6.2 用户提交触发

**业务场景**：用户在备案列表、半年报列表等业务模块中点击"提交审核"按钮，触发流程启动。

**实现机制**：

```
用户点击"提交审核"
    │
    ▼
业务模块调用 BpmProcessInstanceApi.createProcessInstance()
    │
    ▼
BPM 创建流程实例，关联业务数据
    │
    ▼
更新 bpm_business_process 表（业务类型、业务ID、流程实例ID）
    │
    ▼
前端列表显示"待审核"状态，操作按钮变为"查看流程"
```

**表结构支持**：

| 表名 | 作用 |
|------|------|
| `declare_filing` | 备案业务数据 |
| `bpm_business_process` | 业务与流程实例关联 |
| `bpm_business_type` | 业务类型与流程定义关联 |

**DSL 配置示例**：

```json
{
  "nodeKey": "Activity_hospital_submit",
  "cap": "FILL",
  "actions": [
    {
      "key": "submit",
      "label": "提交审核",
      "bizStatus": "SUBMITTED",
      "bizStatusLabel": "已提交",
      "bpmAction": "complete"
    }
  ],
  "assign": {
    "type": "START_USER"
  },
  "enable": true
}
```

### 6.3 定时器触发

**业务场景**：根据填报周期要求，半年报、年报等需要定时自动触发流程。

**实现机制**：

```
定时任务扫描（JobScheduler）
    │
    ▼
检查是否到达触发时间点（如每年6月30日、12月31日）
    │
    ▼
查询需要触发的业务数据（未触发过/草稿状态）
    │
    ▼
对每条业务数据调用 BpmProcessInstanceApi.createProcessInstance()
    │
    ▼
BPM 创建流程实例，发送通知给项目单位
```

**BPM 节点支持**：

| 节点类型 | 说明 | 支持状态 |
|----------|------|----------|
| DELAY_TIMER_NODE | 延迟器节点 | ✅ 支持固定时长 |
| | | ✅ 支持固定日期 |

**定时器配置示例**：

```json
{
  "nodeKey": "Activity_timer_trigger",
  "type": 14,
  "name": "定时触发节点",
  "delaySetting": {
    "type": 2,
    "fixedDateTime": "2026-06-30T00:00:00"
  }
}
```

### 6.4 各流程触发方式汇总

| 序号 | 流程名称 | 触发方式 | 触发条件 | 业务列表显示 |
|------|----------|----------|----------|--------------|
| 1 | 备案流程 | 用户提交 | 用户点击"提交审核" | 提交审核 |
| 2 | 半年报流程 | 定时器 | 每年6月30日 | 自动触发 |
| 3 | 年报流程 | 定时器 | 每年12月31日 | 自动触发 |
| 4 | 中期评估流程 | 用户提交 | 用户点击"提交自评" | 提交审核 |
| 5 | 验收流程 | 用户提交 | 用户点击"申请验收" | 提交审核 |
| 6 | 整改流程 | 审批触发 | 审批人驳回并要求整改 | 待整改 |
| 7 | 转化流程 | 自动触发 | 验收通过后自动触发 | 自动汇集 |

### 6.5 业务状态与流程联动

**状态映射机制**：

```
┌─────────────────────────────────────────────────────────────┐
│                      业务数据状态                            │
├─────────────────────────────────────────────────────────────┤
│  草稿(DRAFT) → 已提交(SUBMITTED) → 审核中(IN_PROGRESS)    │
│           ↓                                   ↓              │
│        退回(RETURNED)                     通过/拒绝         │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│                      流程节点状态                            │
├─────────────────────────────────────────────────────────────┤
│  发起人节点 → 审批人节点1 → 审批人节点2 → ... → 结束节点 │
└─────────────────────────────────────────────────────────────┘
```

**列表按钮显示逻辑**：

| 业务状态 | 列表按钮 |
|----------|----------|
| 草稿 | 提交审核 |
| 已提交/审核中 | 查看流程 |
| 退回 | 重新提交 |
| 已通过 | 查看结果 |
| 已拒绝 | 查看原因 |

## 七、结论与建议

### 7.1 整体评估

| 指标 | 评估结果 |
|------|----------|
| 核心流程支持度 | 5/5 (100%) |
| 节点类型支持度 | 13/13 (100%) |
| 分配策略支持度 | 15+ 种 (100%) |
| 流程触发支持度 | 2/2 (100%) |
| 轻微缺口 | 5 项 |

### 7.2 结论

1. **所有5个核心审批流程均可在现有BPM模块中实现**
2. **流程触发机制完善**：支持用户提交触发和定时器触发两种方式
3. **主要审批能力完全支持**：发起人、审批人、办理人、条件分支、路由分支等
4. **角色配置完善**：支持角色、部门、岗位、用户、用户组、表达式等多种策略
5. **无需进行重大功能扩展**，现有系统已具备完善的支持能力

### 7.3 后续建议

1. **短期**：可直接使用现有能力配置5个核心审批流程
2. **中期**：新增整改任务管理模块，支持任务分解和进度跟踪
3. **长期**：封装评分规则、连续多级审批等高频使用场景的模板
